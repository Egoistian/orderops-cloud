# OrderOps Cloud 포트폴리오 사례 연구

## 한 줄 설명

주문 상태 변경이 권한·회사 경계·감사 이력과 함께 안전하게 처리되는 과정을 보여 주는 멀티테넌트 주문 운영 데모입니다.

## 프로젝트 성격

OrderOps Cloud는 개인 포트폴리오 프로젝트입니다. 화면에 표시되는 회사, 사용자, 주문, 금액, 운영 이력은 모두 합성한 데모 데이터입니다. 실제 고객사 도입, 매출, 운영 비용 절감, 처리 속도 향상 같은 실적을 주장하지 않습니다.

## 문제를 어떻게 정의했나

주문 관리 화면은 표와 통계 카드만 있어도 그럴듯해 보일 수 있습니다. 하지만 실제 업무 도구에서 더 어려운 부분은 다음 질문에 답하는 일이라고 보았습니다.

- 사용자는 자기 회사 데이터만 볼 수 있는가?
- 열람 전용 사용자가 화면을 우회해 변경 API를 직접 호출하면 막히는가?
- 두 명이 같은 주문을 동시에 수정할 때 마지막 요청이 앞선 작업을 덮어쓰지 않는가?
- 주문 상태와 “누가, 왜 바꿨는지”가 서로 어긋나지 않는가?
- 화면이 아니라 서버와 데이터베이스가 이 규칙을 보장하는가?

그래서 디자인 시안에 머무르지 않고 로그인부터 PostgreSQL 저장, 권한 검증, 상태 전이, 감사 로그까지 하나의 동작 경로로 구현하는 것을 목표로 삼았습니다.

## 핵심 사용자 흐름

1. 리뷰어가 합성 데모 계정 중 회사와 역할을 선택해 로그인합니다.
2. API가 세션의 회사 ID로 주문과 운영 지표를 조회합니다.
3. 사용자가 주문을 선택하면 현재 상태에서 가능한 다음 단계만 확인할 수 있습니다.
4. 변경 요청에는 화면이 마지막으로 본 주문 버전과 변경 사유가 함께 전송됩니다.
5. 서버가 회사, 역할, 상태 전이, 버전을 검증합니다.
6. PostgreSQL 트랜잭션 안에서 주문과 감사 이벤트를 함께 기록합니다.
7. 화면은 갱신된 주문과 최신 감사 이력을 보여 줍니다.

## 설계 결정

### 회사 ID를 브라우저 입력으로 받지 않기

보호된 API는 요청 본문에 담긴 회사 정보를 신뢰하지 않습니다. 서버가 `HttpOnly` 세션 쿠키를 조회해 사용자와 회사 컨텍스트를 복원하고, 그 회사 ID를 모든 업무 쿼리에 적용합니다. 다른 회사 주문을 변경하려 하면 존재 여부까지 감추기 위해 `404`를 반환합니다.

### 화면의 비활성화와 서버 권한 검증을 분리하기

클라이언트가 버튼을 숨기는 것은 사용성을 위한 처리일 뿐 보안 경계가 아닙니다. `admin`, `operator`, `viewer`의 전이 규칙을 서버에서도 다시 확인합니다. 따라서 열람 전용 계정이 API를 직접 호출해도 변경은 `403`으로 거부됩니다.

### 상태 변경과 감사 로그를 한 트랜잭션에 넣기

주문만 바뀌고 감사 로그가 빠지거나, 로그만 생기고 주문이 롤백되는 상태를 피하려고 두 쓰기를 같은 트랜잭션에 묶었습니다. 변경 대상 행은 `FOR UPDATE`로 잠그고 성공할 때만 함께 커밋합니다. 감사 이벤트에는 당시 담당자 이름과 역할을 스냅샷으로 남기며, PostgreSQL 트리거가 일반적인 수정·삭제를 거부합니다.

### 변경 요청의 출처와 입력 형식 확인하기

로그인·로그아웃·상태 변경은 `SameSite=Strict` 쿠키에 더해 전용 요청 헤더와 Fetch Metadata를 검사합니다. 로그인과 상태 변경은 JSON 본문만 받고, 잘못된 필터·식별자·본문 크기를 안정적인 오류 코드로 구분합니다. 이는 로컬 포트폴리오의 방어 계층이며 완전한 보안 인증을 뜻하지 않습니다.

### 낙관적 버전으로 덮어쓰기 막기

각 주문에 증가하는 `version`을 두었습니다. 클라이언트의 `expectedVersion`이 DB의 현재 버전과 다르면 `409 VERSION_CONFLICT`를 반환합니다. 자동 병합을 추측하지 않고 최신 정보를 다시 읽도록 하는 단순하고 설명 가능한 선택입니다.

### 데모 데이터도 개인정보처럼 보이지 않게 만들기

시드 데이터는 실존 주문을 복사하지 않았고 이름과 연락처를 마스킹된 합성 값으로 구성했습니다. 두 개의 가상 회사와 세 역할을 제공해 회사 격리와 권한 차이를 재현할 수 있게 했습니다.

## 구현 범위

- React + TypeScript 운영 화면
- Node.js 24 + Express API
- PostgreSQL 16 스키마와 합성 시드
- `scrypt` 비밀번호 해시 검증
- 해시만 DB에 저장하는 8시간 세션 쿠키
- 같은 출처 변경 요청 표식, Fetch Metadata 검사, 보안 응답 헤더
- 회사별 주문·지표·감사 이력 조회
- 역할 기반 주문 상태 전이
- 트랜잭션 기반 주문/감사 로그 동시 기록
- 담당자 스냅샷과 DB 트리거 기반 append-only 감사 이력
- 주문 버전을 이용한 동시 수정 충돌 처리
- 선택형 공개 데모 모드의 tenant 시드 초기화와 자유 입력 메모 치환
- 공개 데모 로그인/reset 및 상태 변경의 인스턴스 단위 요청 제한
- 단위 테스트, PostgreSQL 통합 테스트, 타입 검사, 프로덕션 빌드 CI
- 로컬 데모용 Docker Compose와 OpenAPI 문서

## 검증 방법

이 저장소의 주장은 아래 재현 가능한 검사 범위 안에서만 합니다.

```bash
npm test
npm run test:integration
npm run typecheck
npm run build
```

통합 테스트는 실제 PostgreSQL 스키마를 사용해 다음 경로를 확인합니다.

- 운영 담당자의 상태 변경과 감사 로그가 함께 저장되는지
- liveness와 PostgreSQL readiness가 구분되는지
- 같은 출처 표식 없는 변경 요청이 인증 전에 거부되는지
- 열람 전용 계정의 변경 요청이 서버에서 거부되는지
- 다른 회사 주문 변경이 차단되는지
- 다른 회사 주문의 감사 이력도 `404`로 숨겨지는지
- 오래된 주문 버전이 동시 수정 충돌로 처리되는지
- 감사 이벤트의 일반 수정·삭제가 DB에서 거부되는지

테스트 통과는 위 시나리오의 재현 결과이며, 운영 환경의 성능·보안 인증·가용성을 의미하지 않습니다.

## 현재 한계와 다음 단계

현재 버전은 포트폴리오용 핵심 경로에 집중했습니다. 비밀번호 재설정, MFA, 여러 서버 인스턴스가 공유하는 요청 제한, 페이지네이션, PostgreSQL RLS, 버전형 마이그레이션, 실제 택배사 연동, 웹훅 재시도, 관측성 플랫폼, 백업 복구 훈련은 포함하지 않았습니다. 현재 로그인 실패와 공개 데모 변경 제한은 한 Node.js 인스턴스 안에서만 동작합니다. 공개 모드의 주문 상태도 세션별 복사본이 아니라 tenant별 공유 상태라 다른 방문자의 로그인 초기화와 충돌할 수 있습니다.

다음 단계로 확장한다면 우선순위는 다음과 같습니다.

1. PostgreSQL RLS와 권한 회귀 테스트로 회사 격리를 한 단계 더 강화합니다.
2. 버전형 마이그레이션과 비파괴 시드 전략을 도입합니다.
3. 공유 저장소나 엣지 방화벽 기반 로그인 제한, 세션 관리 화면, 배포 환경별 origin 검증을 추가합니다.
4. 커서 페이지네이션과 필터 인덱스를 실제 데이터 분포로 검증합니다.
5. 요청 ID, 구조화 로그, 오류 추적, 백업 복구 절차를 운영 환경에 맞게 연결합니다.

## 이 작업으로 보여 주고 싶은 역량

화면을 빠르게 만드는 능력뿐 아니라 “누가 어떤 데이터를 어떤 순서로 바꿀 수 있는가”를 API, 데이터베이스 제약, 트랜잭션, 테스트로 연결해 설명하고 검증하는 역량을 보여 주는 프로젝트입니다.
