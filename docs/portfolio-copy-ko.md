# OrderOps Cloud 포트폴리오 소개 문구

아래 문구는 개인 포트폴리오, 위시캣·프리모아 프로필, 프로젝트 상세 페이지에 맞춰 길이별로 사용할 수 있습니다.

## 프로젝트 라벨

**개인 프로젝트 · 합성 데이터 기반 B2B 주문 운영 데모**

## 한 줄 소개

회사별 데이터 격리, 역할 권한, 동시 수정 충돌, 감사 로그까지 구현한 주문 운영 웹 애플리케이션입니다.

## 짧은 소개

OrderOps Cloud는 주문 현황을 보여 주는 화면을 넘어, 운영 담당자가 주문 상태를 안전하게 변경하는 전체 경로를 구현한 개인 프로젝트입니다. React·Express·PostgreSQL로 로그인, 회사별 데이터 격리, 역할 기반 상태 전이, 낙관적 동시성 제어, 트랜잭션·append-only 감사 로그를 구성했습니다. 표시되는 회사·주문·금액은 모두 합성 데이터입니다.

## 상세 소개

주문 관리 도구를 제작할 때 화면 구성뿐 아니라 실제 업무 규칙이 서버와 데이터베이스에서 보장되는지를 보여 주는 데 집중했습니다. 사용자는 가상의 두 회사와 관리자·운영 담당자·열람 전용 역할로 로그인할 수 있습니다. 보호된 API는 세션에서 회사 컨텍스트를 복원하고 모든 주문 쿼리를 해당 회사로 제한합니다.

주문 상태 변경 시 서버가 역할과 상태 전이를 다시 검증하며, 화면이 마지막으로 확인한 주문 버전이 오래되었으면 `409` 충돌을 반환합니다. 정상 변경은 PostgreSQL 트랜잭션 안에서 주문 버전 증가와 감사 이벤트 기록을 함께 커밋하고, DB 트리거는 일반적인 감사 이력 수정·삭제를 거부합니다. 단위 테스트와 실제 PostgreSQL 통합 테스트로 권한 거부, 회사 격리, 상태 변경·감사 기록, 동시 수정 충돌을 재현합니다.

이 프로젝트는 실제 고객사 구축 사례나 운영 성과를 주장하지 않는 개인 포트폴리오입니다. 대신 요구사항을 데이터 모델, API 경계, 권한, 트랜잭션, 테스트, 실행 문서로 연결하는 구현 역량을 보여 줍니다.

## 담당 범위 표기

- 요구사항 정의와 핵심 사용자 흐름 설계
- React + TypeScript 운영 화면 구현
- Express API 및 세션 인증 구현
- PostgreSQL 테이블·제약·인덱스 설계
- 역할 기반 주문 상태 전이 구현
- 주문 변경과 감사 로그의 트랜잭션 처리
- 담당자 스냅샷과 append-only 감사 이력 설계
- 낙관적 동시성 제어와 오류 계약 설계
- 단위·통합 테스트 및 CI 구성
- Docker Compose, OpenAPI, 아키텍처·운영 문서 작성

## 기술 스택

`React` · `TypeScript` · `Vite` · `Node.js 24` · `Express` · `PostgreSQL 16` · `Docker Compose` · `GitHub Actions`

## 의뢰처 관점의 핵심 포인트

- 회사별 데이터가 섞이지 않도록 인증 컨텍스트와 SQL 조건을 함께 설계했습니다.
- 버튼을 숨기는 데 그치지 않고 서버에서 관리자·담당자·열람 권한을 검증했습니다.
- 동시에 수정된 주문을 조용히 덮어쓰지 않도록 버전 충돌을 명시적으로 처리했습니다.
- 주문 변경과 작업 이력이 서로 어긋나지 않도록 하나의 DB 트랜잭션으로 저장했습니다.
- 변경 이력의 일반 수정·삭제를 DB 트리거에서 거부하고 당시 담당자 정보를 스냅샷으로 보존했습니다.
- API 계약, 실행 방법, 한계까지 문서화해 다른 개발자가 검토하고 재현할 수 있게 했습니다.

## 면접·상담에서 설명할 순서

1. “주문표”가 아니라 “안전한 상태 변경”을 핵심 문제로 잡은 이유를 설명합니다.
2. 세션에서 tenant를 결정해 다른 회사 ID를 브라우저가 주입할 수 없게 한 흐름을 보여 줍니다.
3. 열람 계정의 직접 API 호출이 `403`, 다른 회사 주문이 `404`가 되는 통합 테스트를 보여 줍니다.
4. `expectedVersion`과 `409`가 동시 수정 덮어쓰기를 막는 과정을 설명합니다.
5. 주문 갱신과 감사 이벤트가 한 트랜잭션에서 커밋되는 코드를 보여 줍니다.
6. 아직 없는 운영 기능과 다음 단계도 함께 설명합니다.

## 피해야 할 표현

다음 표현은 현재 저장소가 증명하지 않으므로 사용하지 않습니다.

- “실제 고객사 운영 중” 또는 “상용 서비스 출시”
- “매출 ○% 증가”, “처리 시간 ○% 단축”, “비용 ○원 절감”
- “대규모 트래픽 검증”, “무중단 운영”, “엔터프라이즈급 보안”
- “완벽한 멀티테넌시” 또는 “보안 취약점 없음”

대신 “합성 데이터로 재현한 개인 프로젝트”, “통합 테스트로 확인한 시나리오”, “현재 구현 범위”처럼 증거 범위를 함께 적습니다.
